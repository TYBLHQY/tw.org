# 如何请求功能

我们鼓励你请求功能。

你可以使用我们的[问题跟踪系统](https://github.com/GothenburgBitFactory/taskwarrior/issues)提交功能请求。
只需创建一个账户，创建一个问题，并根据你的判断将类型设置为"improvement"或"feature"。
但这不仅仅是这样。
毕竟，你不是想看到你的请求被实现吗？

## 什么是好的功能请求？

请求功能很容易，但你必须记住这并不意味着它会被完成。
它可能是一个简单或模糊的请求。
它可能是一个需要分析的复杂请求，我们会先讨论它。
有时我们甚至会在做任何工作之前写出一个 RFC。
以下是提出好的功能请求的一些提示：

- 描述一个引人注目的功能，让我们真的*想要*它。
  我们发现最有说服力的是一个说明拥有新功能意味着什么或将实现什么的陈述。
  如果你有这个功能，你能做什么新的事情？
  拥有这个功能如何使你更有效地管理任务？
- 描述该功能如何可能与现有功能交互。
  我们更喜欢新功能不仅仅是存储新数据供以后检索的情况 - 我们有 [UDAs](../udas/) 用于此。
- 描述任何可能需要的新报告。
- 描述谁可能从这个功能中受益。
- 描述一个不会对选择不使用它的用户造成成本的功能。
  我们希望基本用法（添加、列表、完成）保持简单和有效。

要记住的一件事是，我们有一个长期的目标是简化我们管理任务的方式，简化 Taskwarrior，并改进一致性。

## 要避免的事情

- 如果你的建议影响基本用法（添加、列表、完成），那么它将不会被实现。
- 不要费力发明语法或想出键盘上未使用的字符。
  我们宁愿你把精力放在思考功能和优势上。
  语法是我们的问题。
  专注于让他人想知道他们如何能够没有这个功能。
- 不要费力建议将 Taskwarrior 引离其预期目的 - 管理任务 - 的功能。
  我们无意向 Taskwarrior 核心添加时间跟踪。
- 如果你有一个可以实现为钩子脚本或附加组件的建议，请改为这样做。
  如果结果足够引人注目，我们可能想将功能迁移到内核，但这不太可能。
- 不要告诉我们某事有多容易，或你能多快添加它。
  实现的难度与想法是否好无关。
  相反，说服每个人这个想法是伟大的。

## 一个好的例子

我认为依赖关系将是一个很好的功能，因为它可以帮助我组织任务并知道我需要先处理哪些。
它还会告诉我现在无法处理的。
被依赖的任务可能会影响"next"报告，因为这些任务需要首先完成。
理想情况下，一个任务可能依赖于多个其他任务。
我需要添加和删除依赖关系。
依赖关系是一个常见理解的概念，因此对许多用户有用。

## 另一个好的例子

我应该被允许在报告中使用"status"字段。
隐瞒只是这一个字段是没有意义的。
为了保持一致性，请将其添加为可报告、可排序的字段。

## 一个不好的例子

我们为什么不能让 @ 字符表示...？
那能有多难？
