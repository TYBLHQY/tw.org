# 有效使用日期

任务不需要截止日期，可以简单地陈述需求：

    $ task add Send Alice a birthday card

但是，这种任务非常适合设置截止日期，也许还可以设置其他几个日期。

有几个日期可以装饰任务，每个日期都有自己的含义和效果。
您可以选择使用其中的一些、全部或都不使用，但与所有 Taskwarrior 功能一样，如果您的需求需要，它们就在那里，但您不会因为不使用它们而付出性能或摩擦的代价。

## `due`（截止日期）

使用 `due` 日期指定任务必须完成的确切日期。
这对应于任务可以被认为按时完成的最后可能时刻。
在我们的示例中，我们可以将 `due` 日期设置为 Alice 的生日
（为了清楚起见，添加了换行符）：

    $ task add Send Alice a birthday card \
               due:2016-11-08

现在您的任务有了关联的 `due` 日期，帮助您确定何时需要处理它。
但这对 Taskwarrior 有什么影响？如何最好地利用它？

我们将任务的 `due` 日期称为"元数据"。
因此，它只是与任务关联的一条数据，因此可以成为过滤器的一部分：

    $ task due:today list
    ...

这是找出任何任务是否今天到期的一种方法。
您也可以使用：

    $ task +TODAY list
    ...

这是一个虚拟标签 `TODAY` 的示例，它不是真实标签，而是您可以查询的东西，等同于前面的示例。
此外，您可以使用 `DUE`，它过滤具有今天或接下来 7 天内截止日期的任务，或使用 `WEEK` 过滤当前日历周内到期的所有任务，或 `YESTERDAY`、`TOMORROW`、`MONTH` 和 `YEAR`。

请注意，任务被视为 `DUE` 的天数可以使用 `rc.due` 设置进行配置。

您可以找到具有任何截止日期的任务：

    $ task due.any: list
    ...

或没有截止日期的任务：

    $ task due.none: list
    ...

还有一个 `overdue`（逾期）报告，它使用 `OVERDUE` 虚拟标签，向您显示已经逾期的任务。
如果您运行 `calendar`（日历）报告，您的截止日期将在其上突出显示。

我们在这里看到的是 Taskwarrior 利用元数据来驱动各种功能。
多个报告将按 `due` 日期排序，如上所示，具有截止日期的任务现在属于您的日程。

## `scheduled`（计划日期）

`scheduled` 日期与 `due` 日期不同，代表处理任务的最早机会。
让我们继续上面的同一示例。
您需要给 Alice 寄一张生日卡，但她的生日要到 11 月才会到来，所以这不是可以提前完成的那种任务。
理想情况下，应该在 `due` 日期前几天完成：

    $ task add Send Alice a birthday card \
               due:2016-11-08 \
               scheduled:2016-11-04

这意味着您需要给 Alice 寄一张生日卡，最晚不超过
2016-11-08，最早不早于 2016-11-04。

如果任务有 `scheduled` 日期，一旦该日期过去，该任务被视为就绪，并且有一个 `ready` 报告和一个 `READY` 虚拟标签：

    $ task ready
    ...
    $ task +READY list
    ...

没有 `scheduled` 日期的任务总是被认为已就绪。
同样，元数据推动了您的任务列表的复杂性。

## `wait`（等待日期）

许多人不喜欢看很长的任务列表，发现它们令人生畏或只是令人分心。
您可以为任务添加 `wait` 日期，其作用是将任务隐藏到该日期。
在我们的示例中，Alice 的生日还没有到来，所以我们应用了 `scheduled` 日期来指示我们还不应该开始该任务，因为它还没有准备好。
现在让我们为任务添加 `wait` 日期：

    $ task add Send Alice a birthday card \
               due:2016-11-08 \
               scheduled:2016-11-04 \
               wait:november

在这里，任务被赋予 `wait` 日期 2016-11-01，通过有用的快捷方式
"november"，这意味着该任务在 11 月之前不会出现在列表中。
那时，它将重新出现，但直到 2016-11-04 它仍然不会准备好。

您可以使用 `waiting` 报告查看所有隐藏的等待任务：

    $ task waiting
    ...

有一个 `WAITING` 虚拟标签来选择这些任务，但请注意您必须将 `all` 报告与其一起使用，否则您会与指定"pending"状态的其他报告发生冲突，因为等待任务不是待处理的。

## `until`（过期日期）

现在假设我完全错过了 Alice 的生日。
真可惜。
任务会逾期，但这种任务我不想迟到完成，我宁愿忘记它，亲自祝 Alice 生日快乐（虽然晚了）。
我可以简单地删除或完成任务，但还有另一个选项，即添加 `until` 日期：

    $ task add Send Alice a birthday card \
               due:2016-11-08 \
               scheduled:2016-11-04 \
               wait:november \
               until:2016-11-10

这意味着在 2016-11-10 上，任务自动删除。
这对于生日卡任务可能是正确的做法，但不适合"支付租金"任务。
小心！

有一个基于 DOM 的快捷方式可以使用，以使上述命令更加规范化：

    $ task add Send Alice a birthday card \
               due:2016-11-08 \
               scheduled:due-4d \
               wait:due-7d \
               until:due+2d

这仅在任务创建时评估，因此如果您更改截止日期，您还需要更改其他日期。
请注意，有一个 `UNTIL` 虚拟标签可以向您显示所有设置为自动过期的任务。

## 其他日期

还有其他与任务关联的日期，但这些主要用于内部使用，对您的用处较少。

每个任务都有一个 `entry` 日期，记录创建时间。
每个已完成或已删除的任务都有一个 `end` 日期，记录完成或删除的时间。活跃或已启动的任务有 `start` 日期，但仅在处于活跃状态时。最后，每个任务都有一个 `modification` 日期，记录上次修改的时间。这在任务同步时用作提示。

此外，您可能会发现为任务列表需要不同类型的日期。例如，您可能遵循敏捷开发过程，任务可能分配给冲刺，该冲刺可能由其结束日期标识。您可以通过定义
[用户定义属性](../udas/) (UDA) 并将该元数据与您的任务存储在一起，为 Taskwarrior 添加这样的任意日期。在这种情况下，Taskwarrior 将对您的 UDA 不做任何操作，只是存储它，并让您在报告和过滤器中使用它。

## 紧急程度

任务中日期元数据的存在和值会影响紧急程度计算。
例如，如果任务被依赖性阻止，紧急程度会降低。
类似地，已准备好的任务会有更高的紧急程度。
