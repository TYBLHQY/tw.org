# burndown.weekly (或 .daily 或 .monthly)

`burndown` 命令显示一个图表，显示不同时间段内待处理、活跃和已完成任务的数量。

此报告可以轻松评估项目的进度，如果在过滤器中指定了项目的话。
使用过滤器，您可以查看任何任务集合的图表。

## 别名

默认情况下有一个名为 `burndown` 的别名，它扩展为 `burndown.weekly`。
这可以被覆盖为显示 `burndown.monthly` 或 `burndown.daily` 报告。

## 收敛

`burndown` 报告试图预测何时所有与过滤器匹配的任务都将完成。
对于版本 {{< label >}}2.5.1{{< /label >}}，使用以下方法：

扫描过滤的任务，以查看是否存在待处理任务的峰值。
如果待处理任务的峰值在过去三天内的任何时间，那么添加的任务多于完成的任务。
在这种情况下，无法预测何时会改变，因此何时所有任务都将完成。

![](../../../images/nofixrate.png)

如果峰值发生在三天多前，并且待处理任务的数量一直在下降，则可以确定*净修复率*，即平均每天完成的任务数。
此速率显示在图表上。

![](../../../images/fixrate.png)

如果已知*净修复率*，则可以确定完成所有剩余匹配任务可能需要多长时间。
此持续时间被添加到当前日期，并可以计算预测的完成日期。
此日期显示在图表上。

![](../../../images/completion.png)

这种方法基于一个常见的模式，即项目往往从计划阶段转向执行阶段。
这简单地意味着任务倾向于在过程的早期创建，在过程的后期完成。
这往往会创建待处理任务的峰值。

基于最近的完成率和计数的收敛日期，估计的完成日期只是一个估计，其准确性是未知的，但与任务大小的一致性、完成任务所花时间的一致性有关。
一种更复杂的方法，如[基于证据的日程安排](https://www.joelonsoftware.com/2007/10/26/evidence-based-scheduling/)（EBS），会考虑估计、实际时间和你的任务历史。

在早期版本中，收敛算法试图太用力，经常出错。
让我们不提及那个。
